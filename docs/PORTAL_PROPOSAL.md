## 1. エグゼクティブサマリー

**Engineer Skill Lab** は、エンジニアが実務で必要とするスキルを「ゲーム感覚で楽しく鍛える」ことに特化した学習アプリポータルサイトです。

ポータルは **8カテゴリ** のタブで整理（`src/types.ts` の `Category`）。`src/data.ts` には **ミニアプリ8本** のメタデータのみを掲載し、不要な題材はリストから除外済みです。現時点で **プレイ可能なのは「クラウドアーキテクチャパズル」1本** のみで、残り7本は `designing` / `planning` として一覧表示されます（**DevOps** カテゴリには現状、掲載アプリなし）。将来的な追加・拡張はロードマップ（セクション8）で計画します。

### 主要KPI目標

| 指標 | 6ヶ月目標 | 12ヶ月目標 | 24ヶ月目標 |
|------|-----------|------------|------------|
| 月間アクティブユーザー | 10,000 MAU | 50,000 MAU | 200,000 MAU |
| 収録アプリ数（実装） | 3本 | 6本 | 8本（`data.ts` 掲載分の実装完了を目安） |
| 平均セッション時間 | 10分 | 15分 | 20分 |
| 月間再訪率 | 40% | 55% | 65% |
| NPS（推奨度） | +30 | +45 | +55 |

## 2. 企画背景と課題

### 2.1 市場背景

エンジニア人材の需要は世界的に拡大を続けており、特にクラウド・DevOps・セキュリティ領域のスキルギャップは年々深刻化しています。リスキリング・アップスキリングの需要は高まる一方、既存の学習手段には構造的な課題があります。

### 2.2 既存の学習手段の課題

| 課題 | 詳細 |
|------|------|
| **学習の断片化** | スキル領域ごとに別のプラットフォームを使い分ける必要があり、学習の一貫性が保てない |
| **受動的な学習** | 動画視聴やテキスト読解が中心で、手を動かす実践機会が乏しい |
| **高い学習コスト** | ハンズオンラボはクラウド利用料が発生し、失敗のコストが学習の障壁になる |
| **モチベーションの欠如** | 達成感やフィードバックが少なく、学習の継続率が低い |
| **スキルの可視化が困難** | 何をどの程度習得したか、体系的に把握する手段がない |

### 2.3 本企画の解決アプローチ

- **ワンストップポータル：** 8カテゴリのスキルを1つのプラットフォームで横断的に学習（実装は `src/App.tsx` のカテゴリタブ）
- **ゲーミフィケーション：** パズル・RPG・対戦・シミュレーションなど多様な形式で能動的な学習を実現
- **ゼロコスト実践：** ブラウザ上で完結し、クラウド利用料やセットアップが一切不要
- **即時フィードバック：** スコア・ランク・模範解答により、学習効果をリアルタイムで実感
- **スキルマップ：** カテゴリ横断の進捗可視化で、自分の強みと弱みを一目で把握

## 3. プロダクト概要

### 3.1 プロダクト構成

```
Engineer Skill Lab（ポータルサイト）— 実装ベース
├── ポータル（src/App.tsx）：検索・カテゴリフィルタ・ソート（おすすめ / 難易度）
├── ミニアプリ（src/phase1/Phase1AppScreen.tsx）：戻るボタン＋アプリ本体の埋め込み
└── 学習アプリ群（src/apps/<appId>/）：現状 cloud-arch-puzzle のみ実装。他IDは Phase1 で未登録
```

**未実装（ロードマップ上の想定）:** スキルマップ、結果シェア画像、ランキング、認証など。

### 3.2 ポータルサイトの役割

ポータルサイトは単なるアプリ一覧ではなく、以下の価値を提供する「学習ハブ」として機能します。

- **発見：** カテゴリ・難易度・タグによる最適なアプリの発見
- **動機付け：** ランキング・バッジ・ストリークによる継続学習の動機
- **体系化：** スキルマップによる学習進捗の体系的な可視化
- **コミュニティ（計画）：** ユーザー同士の学び合い — 現状は未実装

## 6. UI/UXデザイン方針

### 6.1 デザインコンセプト

**「ポップで可愛い」** をベースに、学習のハードルを下げる親しみやすいビジュアルを採用。**ポータル**は Tailwind CSS のユーティリティ＋ Motion でカード・モーダル等を制御。**クラウドアーキテクチャパズル**は同リポジトリ内で独自スタイル（インラインスタイル・CSS）を併用。

### 6.2 デザインシステム

| 要素 | 仕様 |
|------|------|
| 背景 | パステルグラデーション（ピンク→水色→黄→緑の対角線グラデ） |
| フォント（見出し） | Hachi Maru Pop（手書き風・タイトル用） |
| フォント（本文） | M PLUS Rounded 1c（丸ゴシック・本文用） |
| カードスタイル | 白背景・角丸22px・アクセントカラーのトップライン |
| 装飾 | 浮遊する絵文字アニメーション |
| バッジ | ピンク×オレンジのグラデーション（NEW / ソートボタン等） |
| スクロールバー | ピンク系 |
| アニメーション | fadeUp（カード出現）、pop（モーダル）、float（装飾） |

## 7. 技術仕様

### 7.1 ポータルサイト

| 項目 | 技術（`package.json` ベース） |
|------|------------------------------|
| フレームワーク | **React 19**（Hooks） |
| ビルド | **Vite 6** |
| 言語 | TypeScript / TSX |
| スタイリング | **Tailwind CSS 4**（`@tailwindcss/vite`） |
| アニメーション | **Motion**（`motion/react`）、一部 CSS |
| 状態管理 | React `useState` / `useMemo` のみ（Zustand 等は未使用） |
| ルーティング | **なし**（`App.tsx` の `activeAppId` でポータル ↔ ミニアプリを切替） |
| ホスティング | Cloudflare Pages（[デプロイ手順](./CLOUDFLARE_PAGES_DEPLOYMENT.md)、`public/_redirects` で SPA フォールバック） |
| 永続化 | サーバー・ログインなし（クライアント完結） |
| 動画導線 | クラウドアーキテクチャパズル：`content/youtube.json`（`byChallengeId` でチャレンジ別に上書き） |

### 7.2 各アプリのディレクトリ構成（実装例：cloud-arch-puzzle）

```
src/
├── App.tsx                         ← ポータル UI
├── data.ts                         ← 掲載アプリメタデータ
├── types.ts                        ← Category / SkillApp / 難易度・色
├── phase1/
│   └── Phase1AppScreen.tsx         ← 戻るボタン + `APP_COMPONENTS[appId]` 描画
└── apps/cloud-arch-puzzle/
    ├── App.tsx                     ← パズル本体（チャレンジ・採点・キャンバス）
    ├── youtube-entrypoint.tsx      ← ランク別 YouTube 埋め込み
    ├── youtubeTypes.ts
    └── content/
        ├── learning.json
        ├── scenario.json           ← Phase1 ミニ層パズル用
        └── youtube.json            ← デフォルト + byChallengeId 上書き
```

他 `appId` は `src/apps/` に未作成。追加時は `Phase1AppScreen` の `APP_COMPONENTS` に登録が必要。

### 7.3 対応環境

- モダンブラウザ（Chrome, Firefox, Safari, Edge 最新版）
- レスポンシブ対応（PC / タブレット / スマートフォン）

## 10. ビジネスモデル

### 10.1 収益モデル（YouTube視聴）

| 12ヶ月目 | 50,000 | 5% | 4.5 |
| 24ヶ月目 | 200,000 | 7% | 5.5 |

1. **ゲーム×学習の融合：** 問題を解くだけでなく、パズル・RPG・CTF・シミュレーションなど多様な形式で没入感のある学習体験

## 13. リスクと対策

| リスク | 影響度 | 対策 |
|--------|--------|------|
| アプリの品質担保が困難（複数本の開発負荷） | 高 | Phase分けで段階リリース。各Phaseでユーザーテストを実施し、品質を担保してから次へ |
| 特定アプリの利用が偏る | 中 | おすすめ機能のアルゴリズム最適化。不人気アプリは改善 or 統合を検討 |
| 技術トレンドの変化でコンテンツが陳腐化 | 中 | データ駆動設計（サービス定義をJSON化）で迅速な更新。コミュニティ投稿で持続的なコンテンツ生成 |
| 導線CTR/視聴継続が想定を下回る | 高 | 結果画面の動画レコメンド・埋め込みUXを改善し、視聴完了率/リピートを上げる |
| 競合サービスの参入 | 中 | カテゴリ横断・ゲーム性の高さが参入障壁になりうる。先行者優位でコミュニティを構築 |
| 開発リソースの不足 | 高 | コアチーム3名+フリーランス活用。アプリの共通基盤（採点・UI）をライブラリ化し開発効率向上 |

## 14. まとめ

**Engineer Skill Lab** は、「ゲームで楽しくスキルアップ」という明確なコンセプトのもと、エンジニア学習の在り方を変えるポータルサイトを目指します。

**現状（実装ベース）:** ポータル（検索・フィルタ・ソート）と「クラウドアーキテクチャパズル」1本が利用可能。`src/data.ts` には **ミニアプリ8本** のみを掲載（不要題材は削除済み）。未実装7本は `designing` / `planning` として一覧表示。ロードマップは「8. アプリ開発ロードマップ」の計画案と実装進捗を突き合わせて更新する。

### 本企画の3つの強み

1. **ゲーム × 学習の融合** — 座学では得られない「手を動かす楽しさ」
2. **8カテゴリのワンストップ** — エンジニアスキルを横断的にカバーする設計（アプリ本数は段階的に拡張）
3. **ポップで可愛いUI** — 学習のハードルを下げる親しみやすさ（Tailwind CSS + Motion 等）

これらの差別化要素を軸に、個人学習から企業研修まで、エンジニア学習プラットフォームとしての地位確立を目指します。
